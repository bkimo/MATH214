1. Project Overview

An autonomous vehicle must make hundreds of simultaneous safety decisions every second. Each decision is ultimately a logical proposition: "Proceed through the intersection if the traffic light is green AND no pedestrian is in the crosswalk AND the gap to the nearest vehicle exceeds the stopping distance AND the road is not icy." Write that rule incorrectly — a misplaced negation, an ambiguous "or" — and the car may freeze, proceed unsafely, or produce a logical contradiction with another rule in the system.

AV safety standards ISO 26262 ASIL-D mandate formal verification of driving logic before deployment. Formal verification asks: are all rules internally consistent? Is there any configuration of sensor inputs that leaves the vehicle with no valid action (a dead state)? Do two rules ever contradict each other?

The Logic Oracle for Autonomous Vehicles addresses this using a neuro-symbolic architecture: a Large Language Model (LLM) translates natural language driving rules into formal Boolean expressions, and a Reduced Ordered Binary Decision Diagram (ROBDD) performs all logical verification with mathematical certainty. The system also traces decision paths and translates them back into plain English — providing the human-readable audit trail that regulators require.

Why BDDs for AV safety?

Once a driving rule is encoded in a BDD, the graph never makes a logical error. Contradiction checking between two rules takes one BDD AND operation — no enumeration of sensor combinations needed. Dead-state detection requires checking whether any path leads to a no-action terminal. At n=10 sensor variables, the truth table has 1,024 rows; the BDD may have fewer than 20 nodes.

AV logic oracle — system architecture

Core goals

Students will build a system that accomplishes four things in the AV context:

  • Rule encoding — use an LLM to extract Boolean sensor variables and logical operators from AV safety specification prose
  • BDD construction — build an ROBDD for each driving rule via Shannon expansion
  • Safety verification — detect inter-rule contradictions and dead states (sensor combinations that leave the vehicle with no valid action)
  • Audit trail generation — translate BDD decision paths into plain-English explanations for regulatory review

2.3 Why logical errors in AV rules are dangerous

Consider a subtle error in Rule R1: a safety engineer writes "proceed if light is green or no pedestrian is in the crosswalk", using OR instead of AND. In Boolean form: L | ~P instead of L & ~P & G & R. The erroneous rule permits proceeding through a red light whenever the crosswalk is empty — a potentially fatal mistake that looks plausible in prose but is immediately obvious in the BDD (the L=0, P=0 path leads to terminal 1: proceed).

Safety-critical implication

An LLM alone cannot be trusted to detect this class of error — it may produce a valid-looking expression that is logically subtly wrong. The BDD layer provides the correctness guarantee: once encoded, the graph's structure is the ground truth. This is why ISO 26262 ASIL-D requirements mandate formal methods, not informal review.

3. Technical Architecture

3.1 Shannon expansion — applied to AV rules

Rule R1 — Proceed if light is green AND no pedestrian AND safe gap AND road safe — has the Boolean form f = L & ~P & G & R. Applying the Shannon expansion on variable L:

Shannon expansion on R1

f = (L ∧ f|L=1) ∨ (¬L ∧ f|L=0)
= (L ∧ (~P & G & R)) ∨ (¬L ∧ 0)  →  = L & ~P & G & R

When L=0 (red light), the entire function collapses to 0 (Do not proceed) regardless of P, G, and R — the BDD eliminates those nodes entirely. This is the reduction step in action: the ROBDD for R1 at the L=0 branch is a single terminal-0 node, not a full sub-tree.

3.2 Contradiction detection — the critical safety check

Suppose a new rule R6 is added by a different engineer: "Proceed whenever the road is safe and the speed is within limit" — i.e., f₆ = R & S. This rule is intended for highway cruise control but has been accidentally included in the intersection rule set. Computing f₁ ∧ f₆ does NOT collapse to 0, so the rules are not directly contradictory — but R6 allows proceeding through a red light. The Oracle detects this by tracing the R6 path where L=0, R=1, S=1 leads to terminal 1 (proceed), and flagging it as a rule scope violation.

Dead-state detection

A dead state occurs when some combination of sensor readings leads to terminal 0 in every rule in the rule set — the vehicle has no valid action. For an AV, this is a safety-critical failure: the vehicle freezes. Students will implement a check that computes f₁ ∨ f₂ ∨ ... ∨ fₙ over all rules and verifies the result is not constant-0.

Week 1 — Symbolic core (detail)

Students encode all five rules and perform pairwise contradiction checks. The central pyeda workflow is:

# Week 1 — AV rule encoding and contradiction detection
from pyeda.inter import exprvars, expr2bdd
from pyeda.boolalg.bdd import _NODES

# Declare sensor variables — one call per variable, [0] extracts the scalar
L, P, G, R, E, S = [exprvars(v, 1)[0] for v in 'LPGRES']
# Build before expr2bdd: maps variable.uniqid (int) to variable name
uid_to_name = {v.uniqid: v.name for v in [L, P, G, R, E, S]}

# Five AV rules as Boolean expressions
rules = {
    "R1_proceed":   L & ~P & G & R,          # light green, no ped, safe gap, dry road
    "R2_stop_red":  ~L,                        # always stop on red
    "R3_yield_ped": P,                         # yield whenever pedestrian detected
    "R4_emergency": E,                         # pull over if emergency vehicle
    "R5_speed":     ~S,                        # decelerate if speed exceeds limit
}

bdds = {name: expr2bdd(f) for name, f in rules.items()}

# Pairwise contradiction check: f & g == 0?
for n1, b1 in bdds.items():
    for n2, b2 in bdds.items():
        if n1 >= n2: continue
        conj = expr2bdd(rules[n1] & rules[n2])
        # constant-0 BDD: pyeda str() representation is simply '0'
        contradicts = (str(conj) == '0')
        print(f"{n1} ∧ {n2}: {'CONTRADICTION' if contradicts else 'consistent'}")

# Dead-state check: is the union of all rules satisfiable?
union_expr = rules["R1_proceed"] | rules["R2_stop_red"] | rules["R3_yield_ped"]
union_bdd  = expr2bdd(union_expr)
print("Dead state exists:", str(union_bdd) == '0')

Week 2 — Neural bridge (detail)

Students build and evaluate the LLM parser on AV-specific language. AV specifications introduce domain challenges not present in generic logic benchmarks: sensor ranges expressed as inequalities ("gap exceeds stopping distance"), compound conditions with implicit AND, and regulatory shorthand ("ASIL-D compliant condition"). 

# Week 2 — AV-specific LLM parser
import json, re, anthropic
from pyeda.inter import exprvars, expr2bdd

AV_SYSTEM_PROMPT = """You are a formal logic extraction engine for autonomous vehicle
safety specifications. Your ONLY output is a JSON object with keys:
  variables (dict[str,str])  — single uppercase letters mapped to sensor descriptions
  expression (str)           — Boolean expression using & | ~ operators only
  notes (str|null)           — flag any ambiguity or domain assumption

AV DOMAIN RULES:
- 'safe' conditions (gap, speed, road) are True when the condition is SATISFIED.
- 'detected' or 'present' (pedestrian, emergency vehicle) are True when DETECTED.
- 'proceed' conditions are True when proceeding IS PERMITTED.
- Never invent variable names not present in the input text.
- Use explicit parentheses for all compound conditions."""

def parse_av_rule(prose: str) -> dict:
    client = anthropic.Anthropic()
    msg = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=512,
        system=AV_SYSTEM_PROMPT,
        messages=[{"role": "user", "content": prose}],
    )
    return json.loads(msg.content[0].text.strip())

def build_av_bdd(parsed: dict):
    variables, expr_str = parsed["variables"], parsed["expression"]
    used = set(re.findall(r'[A-Za-z]+', expr_str))
    hallucinated = used - set(variables.keys())
    if hallucinated:
        raise ValueError(f"Hallucinated sensor variables: {hallucinated}")
    ns = {name: exprvars(name, 1)[0] for name in variables}
    return expr2bdd(eval(expr_str, {"__builtins__": {}}, ns)), variables

# Example
prose = "Proceed through the intersection only if the traffic light is green, " \
        "no pedestrian is detected in the crosswalk, the gap to the lead vehicle " \
        "exceeds the stopping distance, and the road surface is dry."
parsed = parse_av_rule(prose)
bdd, var_map = build_av_bdd(parsed)
print("AV rule encoded:", parsed["expression"])

Week 3 — Safety oracle (detail)

The full Oracle implements the audit-trail loop required by ISO 26262: for every sensor reading that leads to a STOP or YIELD decision, the system must produce a human-readable justification. Students choose between returning the shortest path (minimum sensor conditions causing the stop) or all paths (full audit of every stopping scenario). 

# Week 3 — AV Safety Oracle: path-finding + audit trail
from collections import deque
from pyeda.boolalg.bdd import BDDConstant
import anthropic

def all_paths_to(bdd, target_val: int, uid_to_name: dict) -> list:
    """Return ALL BDD paths reaching target terminal (0=False, 1=True).
    uid_to_name maps variable.uniqid (int) to name (str) — build before expr2bdd."""
    results = []
    queue   = deque([(bdd.node, {})])
    while queue:
        node, path = queue.popleft()
        # Terminals: root==-1 → True (BDDONE), root==-2 → False (BDDZERO), or None
        if node is None or node.root < 0:
            if node is not None and int(node.root == -1) == target_val:
                results.append(path)
            continue
        name = uid_to_name[node.root]   # node.root is the variable uniqid (int)
        queue.append((node.hi, {**path, name: 1}))   # .hi = True branch
        queue.append((node.lo, {**path, name: 0}))   # .lo = False branch
    return results

def generate_audit_entry(path: dict, var_map: dict, decision: str) -> str:
    """Feed verified BDD path to LLM for regulatory-grade English explanation."""
    client = anthropic.Anthropic()
    msg = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=200,
        messages=[{"role": "user", "content":
            f"AV sensor variable descriptions: {var_map}\n"
            f"Verified sensor state (from BDD path): {path}\n"
            f"Decision reached: {decision}\n"
            "Write one plain-English sentence suitable for an ISO 26262 audit log, "
            "explaining why this AV decision was made. Be precise about which "
            "sensor conditions triggered the decision."
        }]
    )
    return msg.content[0].text.strip()

# ── Demo: enumerate all STOP scenarios for R1 and generate audit entries ──
# Build uid_to_name before expr2bdd so variable uniqids are registered
uid_to_name = {v.uniqid: v.name for v in [L, P, G, R]}
bdd_r1 = expr2bdd(f_r1)
stop_paths = all_paths_to(bdd_r1, target_val=0, uid_to_name=uid_to_name)
print(f"R1: {len(stop_paths)} STOP scenarios found\n")
for i, path in enumerate(stop_paths[:3], 1):  # show first 3
    entry = generate_audit_entry(path, uid_to_name, "STOP/YIELD")
    print(f"Audit [{i}]: {entry}\n")

5.1 System prompt

You are a formal logic extraction engine for autonomous vehicle safety
specifications. Your ONLY output is a JSON object. No explanation.

=== OUTPUT FORMAT ===
{
  "variables": {
    "L": "Traffic light is green",
    "P": "Pedestrian detected in crosswalk",
    ... (single uppercase letters only)
  },
  "expression": "L & ~P & G & R",
  "notes": null
}

=== AV DOMAIN CONVENTIONS ===
- Sensor conditions use positive polarity when SAFE/PERMITTED:
    "safe gap"  → G = True    |  "gap insufficient" → G = False
    "road dry"  → R = True    |  "icy road"         → R = False
    "light green" → L = True  |  "red/yellow light"  → L = False
- Detected hazards use positive polarity when PRESENT:
    "pedestrian detected" → P = True  |  "no pedestrian" → P = False
    "emergency vehicle"   → E = True  |  "no emergency"  → E = False
- Implicit AND: multiple conditions without explicit connector → use &
- "only if" / "when" / "provided that" → all conditions joined with &
- "unless" A → (not A) as a factor in the conjunction

=== RULES ===
- Never invent variable names. Use ONLY letters present in the text.
- Use explicit parentheses for all compound conditions.
- If the rule is ambiguous, set expression to null and explain in notes.

6. Research Questions

Students must document findings for at least two questions. RQ1 and RQ4 are particularly suitable for publication-quality undergraduate research.

01

Nesting depth vs. parser accuracy in AV specs

AV specifications often nest conditions three or four levels deep ("proceed only if… unless… except when…"). At what nesting depth does the LLM parser begin to fail? Design 12 test inputs at depths 1–4, measure accuracy, and plot a hallucination-rate curve. This directly motivates formal verification as a non-negotiable safety layer.

02

BDD compression vs. sensor count

Real AV systems have 10–30 Boolean sensor inputs. Compare BDD node count with 2ⁿ truth table rows for n = 4, 6, 8, 10, 12 sensors. For which rule structures is compression greatest? (Conjunctive rules like R1 compress extremely well; parity-like rules do not.)

03

Dead-state frequency in real AV rule sets

Obtain a published AV operational design domain specification (e.g., Waymo's published safety report or Apollo's rule set). Encode as many rules as possible and check for dead states — sensor combinations where no rule permits any action. How common are they? What is their safety implication?

04

ISO 26262 ASIL-D audit trail quality

ISO 26262 requires that every safety-critical decision be traceable to its triggering conditions. Evaluate the LLM explanation loop against a rubric: (a) does it correctly identify the triggering sensor(s)? (b) is it unambiguous? (c) would a regulator accept it? Score 20 audit entries and report inter-rater agreement.

BDD compression for AV conjunctive rules — node count vs truth table rows
For conjunctive AV rules like R1, BDD node count grows linearly while truth table rows grow exponentially.

Conjunctive rules (all conditions joined by AND) compress dramatically — the BDD is a simple linear chain. Parity-like rules degrade to exponential. Most real AV safety rules are conjunctive and benefit greatly.

7. Real-World AV Safety Context

Industry precedents

The techniques in this project are not academic exercises — they underpin production AV systems:

  • Waymo uses formal specifications (based on temporal logic, an extension of propositional logic) to define and verify Operational Design Domains
  • Mobileye's Responsibility-Sensitive Safety (RSS) model is a formally verified rule system over sensor variables — essentially a large BDD in spirit
  • NASA uses model checkers (SPIN, NuSMV) based on BDD algorithms to verify Mars rover decision logic
  • Toyota's ISO 26262 ASIL-D certification process requires formal verification of safety goal decomposition

Standards and regulation

Your Oracle's outputs map directly onto regulatory requirements:

  • ISO 26262 (Functional Safety) — requires formal verification at ASIL-C/D levels; contradiction detection and dead-state checks are directly applicable
  • SAE J3016 — defines six levels of driving automation; the Logic Oracle applies to levels 3–5 where the system makes all driving decisions
  • UNECE WP.29 — UN regulation requiring AV manufacturers to demonstrate safety of automated driving systems; audit trail generation is a key requirement
The 2016 Tesla Autopilot crash — a logic failure

The May 2016 fatal crash of a Tesla under Autopilot occurred because the system's sensor fusion logic failed to detect a white truck against a bright sky. In Boolean terms: the "obstacle detected" variable was False when it should have been True. Formal verification cannot prevent sensor hardware failures — but it can guarantee that if the sensor variables are accurate, the logical decision is provably correct. This is precisely the contract the Logic Oracle establishes: the BDD layer is a perfect reasoner over its inputs. Sensor reliability is a separate (and equally critical) engineering problem.

Grading note for instructors

The grading criterion is logical soundness of the symbolic layer, not LLM parser perfection. A system where the BDD correctly identifies a contradiction between two rules — even if one rule was manually encoded — scores higher than one where the LLM appears to work but the BDD verification step is omitted. The safety contract must be demonstrably present.
